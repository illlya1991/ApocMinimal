# APOCALYPSE MINIMAL — ДИЗАЙН-ФАЙЛ

## 1. ФИЛОСОФИЯ ПРОЕКТА

**Жанр:** пошаговый симулятор выживания + управление колонией + ролевая игра  
**Перспектива:** бог без рук — игрок влияет, но не управляет напрямую  
**Темп:** 1 игровой день = 1 реальная сессия (5–30 минут)  
**Масштаб:** 365 дней, один город, один Терминал

### Ключевая идея
Игрок не герой. Игрок — условие. NPC — агенты с собственными целями. Игра про то, что происходит когда умные, испуганные, голодные люди сталкиваются с невозможным — и про то, как один нечеловеческий разум пытается направить их, имея всего 1 час в сутки.

### Три кита
1. **Автономия NPC** — они не слушают приказы, они реагируют на условия
2. **Ресурсный реализм** — каждое действие стоит чего-то конкретного
3. **Контроль как процесс** — нельзя «захватить» и забыть. Надо удерживать каждый день.

## 2. АРХИТЕКТУРА ИГРЫ

### 2.1. Главное окно (GameWindow)

```
┌─────────────────────────────────────────────────────────────────┐
│ [Имя игрока] | [Фракция] | День X | ОР: N | Терминал: ур.N | Действий: N/10 │
│ [Игрок] [КВЕСТЫ] [ТЕХНИКИ] [ПОЛНЫЙ ОБЗОР] [⚙] │ КОНЕЦ ДНЯ │
├──────────────┬──────────────────────────────┬───────────────────┤
│              │                              │                   │
│  СПИСОК NPC  │         ЛОГ СОБЫТИЙ          │  ДЕЙСТВИЯ ИГРОКА  │
│  (260px)     │         (растягивается)      │  (310px)          │
│              │                              │                   │
│  NPC1        │  ═══ ДЕНЬ 10 ══════          │  [Действие 1]     │
│  NPC2        │  [Система]                   │  [Действие 2]     │
│  NPC3        │  [NPC1]                      │  ...              │
│  ...         │  [NPC2]                      │                   │
│              │  ...                         │                   │
│  [Sidebar    │                              │                   │
│   NPC info]  │                              │                   │
└──────────────┴──────────────────────────────┴───────────────────┘
```

**Цветовой код лога:**
- `#8b949e` — системные события (нейтральные)
- `#56d364` — позитивные события, успехи
- `#f87171` — опасность, смерти, провалы
- `#f59e0b` — обмены, президентские события
- `#60a5fa` — статус терминала, уровни
- `#79c0ff` — техники, способности

### 2.2. Структура дня

```
06:00–22:59  → NPC действуют автономно (17 часов × до 30 NPC = логи)
23:00–23:59  → Действия игрока (10 очков действий)
00:00        → Конец дня: расчёт ОР, потребностей, квестов, сохранение
```

**Кнопка «Конец дня»** запускает:
1. `AdvanceToNextDay()` — потребности, здоровье, квесты
2. `SaveAll()`
3. `ProcessNpcDay()` следующего дня — логи NPC
4. `SetupAndApplyDayExchanges()` — президентские обмены

## 3. NPC — ДИЗАЙН АГЕНТОВ

### 3.1. Модель принятия решений

```
Приоритет 1: Критическая потребность < 20% → найти ресурс
Приоритет 2: Активный квест → выполнять
Приоритет 3: Взвешенный случайный выбор из доступных действий
```

**Вес действия** = базовый вес × модификатор характера × модификатор эмоций × (1 + random 0–0.3)

### 3.2. Память NPC

Каждое значимое событие записывается в Memory с тегом:
- `Action` — действие NPC
- `Interaction` — взаимодействие с игроком
- `Social` — социальное событие  
- `Combat` — бой
- `Quest` — квест

Память влияет на Trust/Fear и вес действий. NPC с высоким страхом избегает опасных действий. NPC с низким доверием саботирует квесты.

### 3.3. Параметры отношений

| Параметр | Диапазон | Влияние |
|----------|----------|---------|
| Trust | 0–100 | Вероятность выполнения квестов, ОР |
| Fear | 0–100 | Избегание риска, вероятность предательства |
| FollowerLevel | 0–5 | Бонус к статам, лимиты терминала |
| Health | 0–100% | При <20% — NPC не может действовать нормально |

### 3.4. Лимиты последователей по уровню терминала

| Терминал | Lvl0 | Lvl1 | Lvl2 | Lvl3 | Lvl4 | Lvl5 |
|----------|------|------|------|------|------|------|
| 1 | 10 | 5 | 3 | 1 | 0 | 0 |
| 2 | 30 | 10 | 5 | 3 | 1 | 0 |
| 3 | 100 | 30 | 15 | 10 | 3 | 1 |
| 4 | 500 | 100 | 50 | 25 | 10 | 2 |
| 5 | ∞ | 300 | 150 | 50 | 25 | 3 |

## 4. ТЕХНИКИ — ДИЗАЙН СИСТЕМЫ

### 4.1. Жизненный цикл техники

```
Магазин (каталог) → Покупка (ОР) → Инвентарь (копии) → Обучение NPC → LearnedTechIds
```

- Одна копия = один NPC
- NPC не может выучить одну технику дважды
- Техника не возвращается после обучения
- Требования к статам проверяются при обучении

### 4.2. Каталог (1050 техник из БД)

**Общие (50 техник):**
- 2 физических × 10 уровней = 20
- 2 ментальных × 10 уровней = 20
- 1 энергетический × 10 уровней = 10

**Фракционные (1000 техник):**
- 5 фракций × 20 техник × 10 уровней = 1000
- 7 физических + 7 ментальных + 6 энергетических на фракцию

**Именование:** `[Название] [ГенитивУровня]`  
Пример: «Огненный удар Послушника», «Боевой расчёт Мастера»

### 4.3. Режимы активации (ActivationModes)

Каждая техника имеет JSON-список режимов. Количество = min(9, TerminalLevel).

```json
[
  {"ModeType": "CombatVsEnemy", "Accuracy": 0.75, "Damage": 45,
   "CritChance": 0.15, "StatDeps": [1], "ActivationChance": 0.90, "TempEffects": []},
  {"ModeType": "Passive", "IsActive": true,
   "StatDeps": [1], "ActivationChance": 1.0, "TempEffects": []}
]
```

**Типы режимов по типу техники:**

| TechType | Приоритетные режимы |
|----------|---------------------|
| Physical | CombatVsEnemy, CombatSelf, Passive |
| Mental | PeacefulSelf, PeacefulOther, PeacefulArea, Passive |
| Energy | Passive, CombatSelf, PeacefulCraft, PeacefulSelf |

## 5. КВЕСТЫ — ДИЗАЙН

### 5.1. Поток квеста

```
Каталог → Покупка → Библиотека игрока → Активен → NPC берёт → Выполнен/Провален
```

### 5.2. Реакции NPC

| Реакция | Условие |
|---------|---------|
| Выполнить | Trust > 60, потребности удовлетворены |
| Частично | Trust 30–60 или конкурирующая потребность |
| Игнорировать | Trust < 30 или задание слишком опасно |
| Саботировать | Fear > 70 + низкое доверие |
| Предать | Нашёл лучшее предложение от другого ЦС |

### 5.3. Типы квестов в каталоге

- **Добыча ресурсов** — собрать X единиц Y ресурса
- **Разведка** — исследовать локацию
- **Охота** — уничтожить монстра
- **Социальные** — завербовать NPC, улучшить отношения
- **Строительство** — укрепить позицию, создать хранилище
- **Глобальные** — цепочки квестов с нарративом

## 6. ОР (ОЧКИ РАЗВИТИЯ) — ЭКОНОМИКА

### 6.1. Источники

```
Ежедневные NPC:  Level × 2 × (0.5 + Trust/100) × (0.5 + Satisfaction/100)
                 макс 10 ОР/NPC/день

Новый последователь (lvl0): +10 ОР (разово)
Убийство монстра:           +10–100 ОР (зависит от силы)
Сердце монстра:             +10 ОР (сверху)
Зачистка этажа:             +5 ОР
Зачистка дома:              +50 ОР
Выполнение квеста:          по условию квеста
```

### 6.2. Расходы

```
Техника ур.1:         50 ОР
Техника ур.5:         250 ОР
Техника ур.10:        500 ОР
Квест из каталога:    по условию
Повышение терминала:  200 × 5^(уровень-1)
Повышение последователя: индивидуально
```

### 6.3. Баланс прогрессии

**Дни 10–40** (Послушник/Адепт):
- 5–15 NPC, средний Trust ~50 → ~30–100 ОР/день
- Цель: техники ур.1–2, несколько квестов

**Дни 40–100** (Воин/Ветеран):
- 15–50 NPC, Trust растёт → 100–400 ОР/день
- Цель: терминал ур.2–3, техники ур.3–4

**Дни 100–365** (Мастер+):
- 50–300+ NPC → 500–3000+ ОР/день
- Цель: терминал ур.5+, элита последователей

## 7. КАРТА И ЛОКАЦИИ

### 7.1. Иерархия

```
Город → Район → Улица → Дом → Этаж → Квартира/Коммерческое
```

Ровно — реальный город. Улицы, дома и квартиры генерируются процедурно с реальной плотностью застройки.

### 7.2. Типы коммерческих локаций

Магазины, аптеки, рестораны, мастерские, склады, АЗС, больницы, школы, офисы.

### 7.3. Ресурсные узлы

Каждая локация имеет `ResourceNodes: Dictionary<string, double>` — доступные ресурсы. NPC с квестами или самостоятельно посещают локации и добывают ресурсы. Ресурс уменьшается при добыче, восстанавливается медленно или не восстанавливается.

### 7.4. Исследованность

`IsExplored` — была ли локация посещена. Неисследованные локации видны, но NPC не идут туда без квеста или крайней нужды. Опасность неизвестна.

## 8. ТЕХНИЧЕСКИЙ СТЕК

| Компонент | Решение |
|-----------|---------|
| Платформа | WPF, .NET 8, Windows only |
| Архитектура | MVVM (GameViewModel + Code-behind) |
| БД | SQLite через System.Data.SQLite |
| Сериализация | System.Text.Json (полиморфный JSON для ActivationModes) |
| Потоки | BackgroundWorker для LoadingWindow |
| Структура | partial class DatabaseManager (8 файлов) |

### 8.1. Сервисный слой

| Сервис | Назначение |
|--------|------------|
| LocationService | Загружает все локации один раз, Dictionary<int, Location> |
| TechniqueService | Загружает 1050 техник один раз, Dictionary<string, Technique> |

Оба сервиса инициализируются в `LoadingWindow` (BackgroundWorker) и передаются в `GameInitState` → `GameWindow` → `GameViewModel`. Повторной загрузки из БД нет.

### 8.2. Структура БД (шаблон + сохранение)

```
apoc_minimal_template.db  — генерируется MapGenerator (только чтение)
save_1.db / save_2.db ... — копии шаблона при создании новой игры
```

**Ключевые таблицы:**
- `Npcs` — характеристики, LearnedTechIds (JSON), Memory (JSON)
- `Locations` — иерархия, ресурсы, флаги
- `Techniques` — каталог 1050 техник, ActivationModes (JSON)
- `PlayerTechInventory` — (SaveId, TechKey) — одна строка = одна копия
- `Quests`, `QuestCatalog`, `PlayerLibrary`
- `GameLog` — сохранённые логи по дням

## 10. ПОБЕДА / ПОРАЖЕНИЕ

### 10.1. Условия победы

Проверяются каждый `AdvanceToNextDay()`:

```csharp
IsVictory = TerminalLevel >= VictoryTerminalLevel
         && AliveNpcsCount >= VictoryMinFollowers
         && CurrentDay >= VictoryDay
```

### 10.2. Условия поражения

```csharp
IsDefeat = AliveNpcsCount == 0
        || (DevPoints < 0 && дополнительные условия)
```

### 10.3. Экран результата (ResultWindow)

Показывает итог: победа/поражение, день, уровень терминала, живых последователей, краткая сводка прогресса.
